ECS.1 - Amazon ECS Task Definitions Secure Networking and User Configuration

Policy Type: tfpolicy

Detailed Description:

Control ID: ECS.1
Title: [ECS.1] Amazon ECS task definitions should have secure networking modes and user definitions
Source: AWS Security Hub - NIST 800 53 REV5
Resource Type: AWS::ECS::TaskDefinition
Category: Protect > Secure access management
Severity: High

Related Requirements: NIST.800-53.r5 AC-2(1), NIST.800-53.r5 AC-3, NIST.800-53.r5 AC-3(15), NIST.800-53.r5 AC-3(7), NIST.800-53.r5 AC-5, NIST.800-53.r5 AC-6

AWS Config Rule: ecs-task-definition-user-for-host-mode-check
Schedule Type: Change triggered

Parameters:
- SkipInactiveTaskDefinitions: true (not customizable)

Policy Specification:
This control checks whether an active Amazon ECS task definition with host networking mode has privileged or user container definitions. The control fails for task definitions that have host network mode and container definitions of privileged=false, empty and user=root, or empty.

This control only evaluates the latest active revision of an Amazon ECS task definition.

The purpose of this control is to ensure that access is defined intentionally when you run tasks that use the host network mode. If a task definition has elevated privileges, it is because you have chosen that configuration. This control checks for unexpected privilege escalation when a task definition has host networking enabled, and you don't choose elevated privileges.

Note: This control will be retired after February 16, 2026 and removed from all applicable Security Hub CSPM standards. You can refer to the following controls for evaluation of privileged configuration, network mode configuration, and user configuration:
- [ECS.4] ECS containers should run as non-privileged
- [ECS.17] ECS task definitions should not use host network mode
- [ECS.20] ECS Task Definitions should configure non-root users in Linux container definitions
- [ECS.21] ECS Task Definitions should configure non-administrator users in Windows container definitions

Remediation:
For information about how to update a task definition, see "Updating a task definition" in the Amazon Elastic Container Service Developer Guide.

When you update a task definition, it doesn't update running tasks that were launched from the previous task definition. To update a running task, you must redeploy the task with the new task definition.

Clarifications:
1. The policy checks for insecure configurations where host network mode is used WITHOUT explicit secure configurations (either privileged=true OR non-root user specified)
2. The container_definitions field contains JSON-encoded data that must be parsed to access individual container properties (privileged and user fields)
3. If ANY container in a task definition fails the security check, the entire task definition should be flagged as non-compliant
4. Empty or missing privileged/user fields should be treated as insecure when host network mode is enabled

Related Terraform Resources:

1. Resource Type: aws_ecs_task_definition
   Description: Manages a revision of an ECS task definition to be used in aws_ecs_service
   Key Arguments for This Policy:
   - family (Required): A unique name for your task definition
   - container_definitions (Required): A list of valid container definitions provided as a single valid JSON document. This is the primary field to evaluate for security configurations
   - network_mode (Optional): Docker networking mode to use for the containers in the task. Valid values: awsvpc, bridge, host, and none. The policy specifically checks for "host" mode
   - task_role_arn (Optional): ARN of IAM role that allows your Amazon ECS container task to make calls to other AWS services
   - execution_role_arn (Optional): ARN of the task execution role that the Amazon ECS container agent and the Docker daemon can assume
   - requires_compatibilities (Optional): Set of launch types required by the task. Valid values: EC2, EXTERNAL, FARGATE, MANAGED_INSTANCES
   - cpu (Optional): Number of cpu units used by the task. Required if requires_compatibilities is FARGATE
   - memory (Optional): Amount (in MiB) of memory used by the task. Required if requires_compatibilities is FARGATE

   Container Definition Properties to Check (within container_definitions JSON):
   - privileged (Boolean): When this parameter is true, the container is given elevated privileges on the host container instance
   - user (String): The user to use inside the container. Can be specified as a user name, user:group, uid, uid:gid, user:gid, or uid:group

   Note: The container_definitions field is JSON-encoded, so the policy must parse this JSON string to access individual container properties for validation.

Evaluation Logic:
The policy should fail a task definition when ALL of the following conditions are met:
1. network_mode is set to "host"
2. At least one container in container_definitions has:
   - privileged is false or not specified (empty), AND
   - user is "root" or not specified (empty)

The policy should pass when:
1. network_mode is NOT "host", OR
2. All containers in the task definition have EITHER:
   - privileged set to true, OR
   - user set to a non-root value (not "root" and not empty)
